## 0. Task 29.2 brief for Claude Fable

### Mission

Determine the smallest coherent architecture that gives every character a durable, private, and
possibly wrong understanding of the world while reducing, rather than duplicating, the current
memory/context machinery.

Do not implement the candidate design in this document as written. First inspect the live code,
the Task 29.1 baseline artifacts, real prompts, debug evidence, tests, and related open tasks. The
document deliberately contains
competing hypotheses: one localized call, several small sub-agents, a hybrid engine, and an optional
Decision/Prose Narrator split. None is the expected answer.

### Non-negotiable invariants

- The Narrator owns objective physical truth; Character state is subjective belief.
- Human agency, per-session locking, atomic commit, undo, and forward-only schemas remain intact.
- Privacy comes primarily from context selection before model calls, not instructions asking a
  model to ignore already-exposed information.
- Every model call is structured, logged with session/turn/agent identity, and locally validated.
- A new authority must replace overlapping mechanisms; do not leave parallel memories or legacy
  prompt paths.
- Provider cost is not the primary constraint. Correctness, context isolation, and narrative
  quality rank before call count; latency and cache behavior must still be measured.

## 1. Product intent

Give every character a durable, private and potentially incorrect view of the world. The Narrator
continues to own objective reality; each Character receives only what that character believes,
remembers, witnessed, heard, or inferred.

This is not just a fix for an unknown character saying "Oi, Alex". It becomes the canonical home
for evolving subjective information:

- known names, aliases, disguises, and unknown identities;
- relationships, trust, hostility, attraction, and impressions;
- believed life status, condition, and last known location;
- witnessed facts versus hearsay, suspicions, and contradicted beliefs;
- private memories and secrets already represented loosely by `character_notes`;
- knowledge granted by future world-lore, RAG, or Experience features without making it universal.

The design assumes current and future inexpensive capable models. LLM calls are an intentional
semantic layer, while code continues to own privacy, routing, persistence, validation, and atomicity.

> [!IMPORTANT]
> The schemas and turn flow below are candidate designs, not an implementation prescription. The
> first deliverable of this task is an evidence-backed architecture exploration comparing one
> large engine, domain sub-engines, call topology, caching, and asynchronous execution. No runtime
> implementation begins until that exploration freezes the ownership and contracts.

## 2. Subjective State Engine

Treat this feature as a new core engine, provisionally named the **Subjective State Engine**. It is
the single owner of evolving character-relative state and the integration boundary seen by the
Runner. It is not one giant prompt and it is not a bag of unrelated hooks.

### 2.1 Control hypothesis: one localized extra engine

Use one extra core engine with one narrow model-backed operation as the simplest experimental
control, not as the preferred outcome. Compare it against small isolated agents and hybrid
topologies. Its exact position in the turn is deliberately unresolved: it may run before the
Narrator, between the Narrator and Character, or at another boundary discovered by exploration.

The control combines domains in one structured call when their context and authority appear to
match. The experiment must test whether that shared context itself causes leakage, semantic
conflicts, or prompt bloat. Splitting domains into sub-agents is an equal candidate when their
required contexts or authorities differ.

If later evidence earns a sub-agent, it must have:

- one narrowly named responsibility and an explicit stop condition;
- a clean context assembled specifically for that responsibility;
- no conversation history beyond the evidence required for its current task;
- a small provider-enforced JSON schema;
- no authority to save, route, call another agent, or mutate a foreign namespace;
- a stable agent identity in `debug.jsonl`;
- deterministic local validation before its proposal can enter the shared draft.

Possible future jobs are: "does Sofia know C1's name after these events?", "what relationship delta did
this exchange create?", "which life-status claim does Sofia now believe, and from what source?",
and "which old private beliefs still matter after this update?" These are deliberately smaller than
"rewrite everything Sofia knows about the whole world".

Context isolation is the primary control. Prompt rules remain useful inside each job, but the system
must not depend on a giant instruction harness saying "ignore these fields" after already exposing
them. If a sub-agent does not need a fact, code does not send that fact.

Likewise, the final Character call must not receive a raw dump of the complete ledger plus dozens
of injected control variables. A dedicated projection/curation boundary selects a compact,
viewer-safe brief relevant to the current response. Persisted structured state remains the source
of truth; the Character sees only the necessary projection.

The outer engine owns:

- lifecycle: uninitialized, initializing, current, pending update, and failed;
- the canonical private-state schema and per-character namespace isolation;
- selection and scheduling of work for a turn;
- deterministic perception/audience filtering before any model call;
- orchestration of model-backed and deterministic sub-engines;
- validation and deterministic merge of sub-engine operations;
- prompt projection for Narrator, Character, preview, replay, and debug tools;
- transaction participation, revision checks, undo snapshots, and failure semantics;
- cost, latency, cache, and correctness evidence for every internal call.

### 2.2 Candidate internal domains

The exploration must decide the exact boundaries. The initial capability map is:

| Candidate domain | Domain responsibility | Baseline and possible future execution |
|---|---|---|
| Initializer | Compile one character's scenario priors into private initial state | Combined engine call first; split only if initialization needs a distinct context |
| Perception compiler | Turn canonical events into viewer-specific observations | Deterministic filter first; semantic call only for ambiguity |
| Identity resolver | Known names, aliases, introductions, disguises, mistaken identity | Same combined update first; possible isolated job after evidence |
| Relationship engine | Relationship, trust, hostility, attraction, social history | Same combined update first; possible isolated job after evidence |
| Belief/status engine | Claims, confidence, alive/dead/missing beliefs, condition and last location | Same combined update first; possible isolated job after evidence |
| Private memory engine | Durable subjective summary and superseded/important memories | Same combined update or pressure-triggered job, based on measurement |
| Context curator | Select a compact viewer-safe brief relevant to the next Character task | Deterministic selection, with bounded semantic ranking only if needed |
| Prompt projector | Render the curated brief without IDs, unknown names, or foreign state | Deterministic |
| Policy/validator | Audience, provenance, operation limits, namespaces, merge conflicts | Deterministic |

These are state responsibilities, not predetermined Python classes, calls, or agents. The simple
architecture keeps them inside one box. Only a measured failure of context isolation, correctness,
latency, or schema quality justifies turning one into its own mini-box. Any resulting mini-box
returns typed proposals; it never saves `GameState` or calls another mini-box through a hidden side
route.

### 2.3 Orchestration contract

The Runner calls the outer engine at one explicit turn boundary selected by exploration. The outer
engine supplies isolated inputs, validates its structured result, and returns one state delta to the
existing turn draft. Initialization and prompt projection may be methods of the same owner without
becoming additional ordinary-turn model calls.

If a future work graph becomes necessary, it is data, not model improvisation. Code declares its
dependencies; a model cannot create arbitrary jobs, expand its own context, or recursively delegate.

No domain sub-engine may become a second Runner. Session loading, locks, revision, atomic save,
human agency, and final routing remain outside it. Conversely, the Runner must not accumulate
identity/relationship/death heuristics; those belong behind the engine contract.

### 2.4 Candidate staged mini-box architecture

Async fan-out is a competing hypothesis, not a promised implementation. Keep all extra semantic
calls localized inside the outer engine so an internal split does not scatter orchestration across
the core.

If evidence requires mini-boxes, explore a bounded DAG executed in ordered stages:

```text
outer engine
  stage 1: [1] [2] [3]   # independent mini-boxes may run concurrently
             barrier     # await, validate, deterministic merge
  stage 2: [4] [5]
             barrier
  stage 3: [6] [7] [8]
             final delta
```

Start by benchmarking:

1. one monolithic call per character;
2. multiple domain calls per character;
3. one shared event-analysis call followed by isolated per-character projection;
4. hybrid execution: deterministic routing plus only the necessary semantic sub-engines;
5. sequential cache warm-up followed by parallel fan-out versus fully concurrent cold calls.

Do not implement the DAG executor merely to preserve hypothetical flexibility. If selected later,
it has a per-turn concurrency limit, skips jobs with no relevant evidence, and never permits
unbounded `character_count × domain_count` fan-out.

Parallel completion order must never affect persisted output. Results are keyed by
`(character_id, domain)`, sorted into a deterministic merge order, and committed only after every
required result validates. Sibling tasks are cancelled and awaited on failure.

"Asynchronous" must not mean untracked post-commit mutation. The first design should keep required
state changes inside the session transaction. A true background updater is allowed only if the
exploration specifies a durable job record, expected revision, lock reacquisition, stale-result
rejection, cancellation, restart recovery, and replay/debug semantics. Without all of those, it is
out of scope rather than an eventually-consistent side channel.

### 2.5 Placement exploration and optional Decision Layer

The first architectural question is where the one extra engine belongs. Test at least:

1. **before Narrator:** compile or update subjective/canonical decision state before narration;
2. **between Narrator and Character:** consume the existing structured Narrator result, update only
   the selected/affected viewpoints, then project safe context for Character;
3. **around a split Narrator:** only if exploration proves the existing Narrator is too overloaded.

Option 2 is closest to the reported Sofia/Alex bug and adds the smallest new boundary, but it must
not be preselected without tracing the real prompts and state flow.

The broader split-Narrator option introduces a **Decision Layer** box that owns semantic world
decisions while leaving literary rendering to a separate Narration operation. This is a fallback
architecture to study, not the baseline implementation.

If that broader option is selected, start with the simplest useful topology inside it:

1. **one structured Decision call** resolves the attempted action into canonical event proposals,
   state deltas, and the minimum routing/perception evidence required by the rest of the turn;
2. deterministic code validates those proposals against presence, agency, schemas, and current
   state;
3. **one separate Prose Narrator call** renders only the validated decision into reader-facing
   narration;
4. the Subjective State Engine consumes the same validated decision rather than reverse-engineering
   facts from prose.

"Alex saiu da sala" therefore has explicit ownership: the Decision call decides whether the exit
succeeded; code validates/applies a typed presence delta; the subjective engine decides who could
perceive it and how their beliefs change; the Prose Narrator describes the already-decided event.

Do not split the Decision call into identity, relationship, routing, world-resolution, and status
calls on day one. Additional internal agents are earned by observed context pressure, correctness
failures, latency measurements, or genuinely different visibility requirements.

After a validated decision exists, independent work may fan out: prose rendering, per-viewpoint
perspective updates, and a future isolated speaker-routing job can potentially run concurrently.
The Prose call cannot run concurrently with the Decision call because it depends on its validated
result. The work graph must express that dependency instead of treating "async" as universal.

This decomposition is a candidate extension of Task 29.2, not a foregone scope expansion. Claude
Fable must determine whether it belongs in this task, is a prerequisite, or deserves a separate
supertask. The delivered architecture must not leave two competing authorities for world truth.

### 2.6 SDK boundaries around the boxes

Plugins continue to own alternate gameplay such as multiple Character speeches or unusual
roleplay mechanics. The core should expose wrappers at stable semantic operations, not at every
temporary internal sub-agent discovered during experimentation.

The baseline exposes only the stable outer Subjective State operation and preserves the existing
Narrator/Character wrappers where compatible. Additional candidate SDK boundaries are explored
only if the corresponding internal split is selected:

- Subjective State proposal/result boundary scoped to one character namespace;
- Character call wrapper after viewer-safe context projection.
- optional Decision context/schema contribution and complete-call wrapper;
- optional validated Decision result filter before state application;
- optional Prose Narrator context and complete-call wrapper.

A wrapper may replace or surround its logical operation, but the Runner still enforces session
locks, human agency, schema validation, namespace isolation, and atomic commit. Internal agents can
be regrouped later without breaking plugins as long as these outer boxes retain their contracts.
Do not publish domain-level hooks merely because a prototype happens to contain that domain.

## 3. Ownership model

| State | Owner | Truth status | Prompt visibility |
|---|---|---|---|
| `Scene`, bodies, public world history, `story_summary` | Narrator | Objective/canonical | Narrator only |
| Character sheet `mind` | Scenario/core | Initial private identity and priors | That Character + Narrator |
| Subjective perspective ledger | One Character | Belief, possibly false | That Character + Narrator/observer infrastructure |
| Audience/whisper metadata | Runner | Canonical visibility boundary | Programmatic guards |
| Raw provider/debug evidence | Observability layer | Execution evidence | Never injected casually into role prompts |

The Runner never decides narratively whether a belief is true. It decides which events a character
was eligible to perceive, validates structured operations, and prevents canonical names or private
facts from crossing the wrong boundary.

## 4. Candidate forward-only canonical state

Replace `GameState.character_notes` with a typed per-character ledger. A provisional shape is:

```json
{
  "character_perspectives": {
    "C2": {
      "schema_version": 1,
      "initialized_turn": 1,
      "processed_through_turn": 3,
      "self": {
        "memory_summary": "A crowded party has put her in a playful mood."
      },
      "people": {
        "C1": {
          "known_name": null,
          "known_aliases": [],
          "reference": "the unfamiliar man surveying the room",
          "relationship": "stranger",
          "impression": "He seems lost but interesting.",
          "believed_life_status": "alive",
          "believed_condition": null,
          "last_known_location": "living room",
          "confidence": "observed",
          "updated_turn": 1
        }
      },
      "beliefs": {
        "belief-0001": {
          "claim": "The unfamiliar man arrived alone.",
          "source_kind": "observed",
          "source_character": null,
          "source_turn": 1,
          "confidence": "uncertain",
          "superseded_by": null
        }
      }
    }
  }
}
```

The exact fields must be frozen through fixtures before implementation. Required properties of the
final contract:

- maps use stable internal character IDs, but those IDs and unknown canonical names are stripped
  when rendering a Character prompt;
- `known_name` is nullable and may contain an alias or mistaken identity rather than canonical truth;
- observation, attributed report, and inference remain distinct;
- claims can be contradicted or superseded without silently rewriting history into certainty;
- provenance includes the responsible turn;
- arbitrary narrative detail remains possible through bounded text fields;
- unknown fields and invalid enums fail locally;
- no compatibility reader or migration is added.

Because this changes persisted visibility semantics and removes `character_notes`, increment
`SESSION_SCHEMA_VERSION`. Existing sessions remain on disk but are incompatible by design.

## 5. Candidate first-message initialization

Initialization is lazy and transactional on the first observable message, rather than session
creation. Empty sessions therefore cost nothing and can still be edited before play begins.

For each character, run an isolated structured initializer that receives:

- the shared static scenario/cast material needed to resolve references;
- only that character's `mind`, initial knowledge, and narrator directives as subjective priors;
- the rule that missing evidence means unknown, never an invented relationship;
- canonical IDs for machine output, without exposing the result to another Character.

The initializer emits the complete version-1 ledger for that viewpoint. Calls may see canonical
truth as compilation input, but the resulting Character prompt never does. For example, if Sofia's
knowledge does not mention Alex, `people.C1.known_name` must be null even though the initializer can
resolve the internal target as `C1`.

All initializer calls belong to the same session transaction. Invalid output receives the shared
structured-response retry. If a ledger still cannot be initialized, fail the turn without a partial
save or Character call; do not silently fall back to universal knowledge.

## 6. Candidate per-turn perception and belief updates

### 6.1 Narrator output

Replace the single ephemeral `context_for_character` string with typed perception events. The
Narrator remains the correct role to decide what physical event occurred and who could perceive it.
A candidate event is:

```json
{
  "event_kind": "observation",
  "subject_id": "C1",
  "content": "The unfamiliar man is looking around the living room.",
  "witness_ids": ["C2", "C3"]
}
```

Other event kinds should cover audible claims, identity claims, confirmed physical outcomes, and
scene changes. `witness_ids` is validated against presence and whisper audience; a model cannot
grant perception to an absent or excluded character.

The Narrator may use canonical IDs structurally. Character-facing rendering resolves each subject
through that viewer's ledger, so a null `known_name` becomes the private `reference`, never `Alex`
or `C1`.

### 6.2 Perspective updater

After the Narrator call and before any routed Character call, run one isolated updater for every
character receiving new perception events. Inputs contain:

- that character's current ledger;
- only speech/actions/whispers visible to that character;
- only Narrator perception events addressed to that character;
- only that character's private thoughts;
- the unprocessed interval since `processed_through_turn`.

The updater returns bounded operations (`add_belief`, `update_person`, `supersede_belief`,
`rewrite_memory_summary`) rather than replacing another character's state or the complete
`GameState`. Code validates and applies operations to an isolated draft.

Identity learning must be evidence-based. A self-introduction such as "Sou Alex" can set Sofia's
known name; merely receiving an internal `subject_id=C1` cannot. A report that Alex died produces a
reported belief, while witnessing Narrator-confirmed death produces an observed belief. Neither
changes objective world state.

The generated Character response itself becomes input to the next update interval. This avoids a
second full update wave after the Character call while ensuring it is processed before that
viewpoint is used again. If benchmarks show unacceptable stale state for silent witnesses, add a
same-transaction post-response delta only for affected listeners rather than reprocessing everyone.

### 6.3 Character call

The Character receives:

- its stable own mind;
- its rendered private ledger;
- recent visible events with speaker labels resolved through its ledger;
- the current turn's perception events;
- its own previous thoughts.

It never receives raw Narrator prose, `story_summary`, another ledger, canonical IDs, or an unknown
canonical name. `SPEAKER=Alex` therefore becomes a viewer-relative label such as
`SPEAKER=the unfamiliar man` until the identity is learned.

## 7. Prompt caching strategy

Use provider-native prefix caching already verified by the project; do not add an application
response cache or manual cache keys.

Order initializer and updater prompts as follows:

1. stable role rules and JSON contract;
2. immutable scenario/cast block shared by initialization calls;
3. stable viewpoint identity and initial mind;
4. append-only visible/perception event history;
5. current private ledger and newest operation request in the changing suffix.

This yields two useful cache patterns:

- initialization calls share a long static prefix; optionally warm one call before fanning out the
  remaining independent calls if real evidence shows concurrent cold calls cannot reuse it;
- each character's updater forms its own append-only cache chain across turns.

The Decision/Prose split provides another cache hypothesis. Both calls can begin with the same
stable rules and identical world/input context, with the task-specific request placed late. The
Prose request then appends the validated Decision result before asking for narration. The inputs are
not fully identical, but provider prefix caching may reuse everything before that appended suffix.

Do not assume this works merely because the semantic inputs are similar. Cache reuse requires the
serialized message prefix (roles, ordering, text, and provider adaptation) to remain identical. A
different early system prompt destroys the intended shared prefix, and fully concurrent cold calls
may race before any prefix is warm. Compare exact request shapes and real hit/miss evidence before
choosing between sequential warm-up, dependency-driven fan-out, or distinct per-agent cache chains.

Small sub-agents should each have stable instruction prefixes. Cache measurements must be reported
per job family (`perspective:identity:*`, `perspective:relationship:*`, etc.), so a high aggregate
hit rate cannot hide one badly ordered or constantly invalidated sub-agent prompt.

Learning a name or compacting history may cause a legitimate miss from the changed section. It must
not trigger manual invalidation; later calls naturally establish a new reusable prefix. Every call
uses stable labels such as `perspective:init:C2` and `perspective:update:C2`, and `debug.jsonl`
preserves normalized cache hit/miss counters.

Add a controlled DeepSeek/llama.cpp probe for repeated updater calls. The acceptance target is
evidence of prefix reuse, not a hard-coded percentage that would couple correctness to a provider.

## 8. Simplifications and removals

### Remove `character_notes`

The ledger's `self.memory_summary`, people map, and belief entries replace the free-form private
note. Remove the field from `GameState`, Character prompt construction, checkpoint files, preview,
tests, and documentation. Do not maintain both authorities.

### Remove private compaction calls

Continuous perspective updates preserve durable private memory before old records are trimmed.
Compaction keeps the public world summarizer and `story_summary`, but no longer fans out
`summarizer:<character>` calls or rewrites per-character notes. The perspective ledger is current
structured state, like Scene, and is not compacted into a second representation.

Task 23 must be reconciled with this design: public events still compact into `story_summary`, while
private recall is protected by continuous ledger updates rather than late `character_notes`.

### Remove `context_for_character`

Delete the field from Narrator schema, response fixtures, Runner routing, plugins, replay, and
prompt previews once typed perception events are live. No deprecated field or dual-read bridge
remains.

### Tighten confidentiality knowledge

`known_tokens` must stop whitelisting every character ID/name and all scene facts universally.
Known identities and facts come from the viewer's ledger plus deterministically visible records.
Audience checks, whisper redaction, and the Character output guard remain code-owned defense in
depth; they are not replaced by model judgment.

### Keep separate

- `story_summary`: objective Narrator memory;
- Scene/body: objective physical state;
- `mind.knowledge`: immutable scenario priors used to initialize the ledger;
- `current_mood`: keep in its present owner initially; consider merging only after evidence shows a
  single clear ownership model;
- plugin state: remains namespaced and cannot become a shadow perspective store.

## 9. Transaction, undo, compaction, and concurrency

- Initialization and update calls run under the existing per-session turn lock against an isolated
  draft.
- A turn commits history, Scene/mood changes, and perspective operations in one atomic save with one
  revision increment.
- `TurnRecord` or an equivalent turn snapshot gains the pre-turn perspective state needed for exact
  undo. Undo of the first turn returns the session to uninitialized state.
- Compaction checkpoints stop carrying `character_notes`; perspective state is preserved as live
  canonical state. If compaction plugins can mutate it, their deltas must participate in the existing
  checkpoint conflict machinery.
- Delete, fork, snapshot, restore, preview, and replay operate under their current session lock and
  copy the new canonical state exactly.
- No updater endpoint may read or mutate a session outside the Runner transaction.

## 10. Plugin and future-task integration

This boundary belongs in core because every Character prompt depends on it. Plugins may contribute
validated perception/belief proposals through a future narrow hook, but cannot replace privacy or
write arbitrary ledgers from post-commit actions.

- Task 16 Global World Lore remains objective and Narrator-only. A Character learns lore only when
  a witnessed/reported event enters its ledger.
- Task 26's impossible-knowledge and internal-ID findings become first-class acceptance cases.
- Task 23's private-memory goal is absorbed by continuous ledger updates; its public trim/compaction
  gap remains relevant.
- Future RAG retrieves external volume but does not become a second authority for beliefs already in
  the ledger.
- Experiences may provide initial relationship priors through character knowledge/configuration,
  which the initializer compiles into the same canonical state.

### 11.1 Decision scenario matrix

Use scenarios to discover the architecture rather than merely confirm it. For each scenario,
capture objective world state, each viewpoint's expected belief, allowed call inputs, persisted
state, Character-facing projection, and debug evidence. Claude Fable may add or merge scenarios.

| Scenario | Architectural pressure being tested |
|---|---|
| Stranger at the party | Sofia must not learn Alex's canonical name from labels, roster, narration, or context metadata |
| Audible self-introduction | Sofia learns the name in time to use it naturally in her next response |
| Introduction whispered to someone else | Presence alone does not grant the name; audience controls learning |
| False name and later correction | Belief may differ from canonical identity and retain provenance |
| Disguise or mistaken identity | Character-relative identity cannot be a nullable canonical-name field only |
| Reported death | "Someone said Alex died" remains attributed hearsay, not observed or objective fact |
| Witnessed death | Witnesses update differently from absent characters and people outside the view/audience |
| Apparent death followed by sighting | Contradiction changes belief without rewriting what was previously believed |
| Resurrection confirmed by Narrator | Objective state and each character's delayed knowledge remain separate |
| Two witnesses interpret one event differently | Perspective is not just a copy of filtered canonical facts |
| Relationship rupture | Trust/hostility changes from an exchange without forcing every domain to rewrite |
| Secret propagating as rumor | Whisper knowledge, second-hand claims, and public confirmation remain distinct |
| Character leaves and returns | Missed events do not enter private memory through later compaction or summaries |
| Several unknown people present | Stable references distinguish strangers without leaking `C1`/`C2` identifiers |
| Private thought contradicts speech | Only the owner retains the thought; observers update from what they perceived |
| Undo after a belief change | World, history, and every affected viewpoint restore exactly together |
| Fork before contradictory evidence | Forks can retain different beliefs without shared mutable state |
| Compaction after long play | Subjective memory survives trimming without a parallel private-memory authority |
| Five-plus-character turn | Call fan-out, context isolation, latency, cancellation, and cache behavior remain bounded |
| Sub-agent failure mid-stage | No partial belief state commits; sibling work is cancelled and evidenced |

At minimum, compare every surviving architecture on the stranger, whisper, false identity,
reported-versus-witnessed death, absence, undo, long-session, and five-character scenarios. A design
that only solves the initial name leak is insufficient; a design that cannot solve that simple case
without many unrelated calls is also suspect.

## 13. Acceptance suite

### Identity and relationship

- Sofia starts with no Alex reference in her knowledge: Narrator prompt contains `Alex`, Sofia's
  prompt contains neither `Alex` nor `C1`, and her ledger uses an unknown reference.
- "Sou Alex" heard by Sofia updates her known name before her response.
- An introduction whispered outside Sofia's audience does not update her.
- A false alias remains Sofia's belief until contradicted; canonical truth is not injected.
- Multiple unknown characters receive distinct stable subjective references without internal IDs.

### Belief and provenance

- Witnessed death, reported death, suspicion of death, later sighting, and resurrection remain
  semantically distinct and update provenance correctly.
- An absent character does not learn a public-room event it did not witness.
- A character may hold a false belief without the Narrator's truth silently correcting it.
- Another character's thought, ledger, or private memory never appears in the updater request.

### Transactional behavior

- Invalid initializer output aborts the first turn without partial ledgers or history.
- Invalid updater operations never cross character namespaces.
- Ordinary undo restores the exact prior perspectives; undoing turn one removes initialization.
- Concurrent turn/undo/compaction operations serialize under the session lock.
- Fork and restore preserve divergent subjective beliefs independently.
- Old schema sessions return `incompatible_session` and are never migrated.

### Simplification

- No `character_notes` field or private summarizer call remains.
- No Narrator `context_for_character` field or consumer remains.
- Public compaction still preserves world history and undo checkpoints.
- Perspective memory survives history trimming without a second private summary.

### Real-model and cache evidence

- Repeat the party scenario from the reported screenshot with Sofia never introduced to Alex; no
  generated Sofia speech/thought uses his name.
- Repeat with an audible introduction; Sofia can use the name afterward.
- Run at least five characters with mixed presence and a whisper audience over enough turns to
  exercise initialization, updates, identity changes, false reports, undo, and compaction.
- Inspect every `perspective:*`, Narrator, and Character request/response in `debug.jsonl` for
  session/turn/agent metadata and private-boundary compliance.
- Demonstrate non-zero provider-native prefix reuse on repeated perspective updates and retain a
  changed-prefix negative control.
